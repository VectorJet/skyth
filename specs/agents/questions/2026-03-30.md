## Phase 2: Agent Architecture Questions

### Q2.1: Agent Directory Structure
The legacy spec calls for agents in `backend/agents/{agent_name}/`. Current implementation has them in `skyth/agents/{agent_name}/`. 

**Question:** Should we keep current structure or migrate to legacy spec path? Current structure:
- `skyth/agents/generalist_agent/agent_manifest.json`
- `skyth/agents/generalist_agent/agent.ts`
- `skyth/agents/generalist_agent/tools/`

### Q2.2: Specialized Agents Implementation
Legacy spec calls for 3 specialized agents:
- `code_agent` - Code generation, debugging
- `research_agent` - Research, web search
- `data_agent` - Data processing, analysis

**Question:** Should we create these as new agent packages in `skyth/agents/`? Each would have their own manifest with specific capabilities.

### Q2.3: Delegation Tools - `delegate` vs `task`
Current implementation has `SubagentManager.spawn()` for spawning subagents. Legacy spec has two distinct tools:
- `delegate(agent, task, context)` - Call peer agent (horizontal)
- `task(subagent, todo)` - Spawn subagent (vertical)

**Question:** 
1. Should we implement `delegate` as a separate tool from the existing subagent spawning?
2. Current `DelegationCallStack` has maxDepth=2. Is this correct for Generalistâ†’Agentâ†’Subagent hierarchy?

### Q2.4: Agent Manifest Schema
Current manifest has:
```json
{
  "id": "generalist_agent",
  "name": "Generalist Agent",
  "version": "1.0.0",
  "description": "...",
  "global_tools": true,
  "entrypoint": "agent.ts",
  "capabilities": ["chat", "tools", "delegation"],
  "dependencies": [],
  "security": {}
}
```

Legacy spec adds:
- `type`: "generalist|specialized"
- `tools`: ["array", "of", "tool", "names"]
- `subagents`: ["array", "of", "subagent", "names"]
- `max_context_tokens`: number
- `model_preferences`: { primary, fallback }

**Question:** Should we extend the manifest schema to include legacy spec fields?

### Q2.5: MCP-to-TOON Converter
Legacy spec mentions converting MCP tool **outputs** to TOON format (~40% token reduction), but keeping inputs as JSON.

**Question:** Should we implement this converter? It's not present in current codebase.

## Phase 3: Quasar Memory System Questions

### Q3.1: Quasar Implementation Strategy
Legacy spec calls for Rust core with Python bindings via Maturin. Current codebase has:
- `skyth/memory/backend.ts` - Interface definition
- `skyth/memory/backends/static_sqlite.ts` - Basic SQLite backend

**Question:** 
1. Should we implement Quasar in pure TypeScript (simpler, matches current stack)?
2. Or follow legacy spec with Rust core + Python bindings?

### Q3.2: Quasar Layer Implementation Priority
Legacy spec 5 layers:
- Layer 0: Canonical (encrypted JSONL, AES-256)
- Layer 0.5: ChromaDB (vector embeddings)
- Layer 1: JSONL (append-only logs)
- Layer 2: SQLite (relational)
- Layer 3: PostgreSQL + pgvector (optional)
- Layer 4: Redis (deferred)

**Question:** Which layers should we implement first? Current codebase has basic SQLite.

### Q3.3: Event System (Solars/Nebulas)
Legacy spec describes:
- **Quasar events:** All ticks tracked
- **Solars:** User edits create branches
- **Nebulas:** Regenerations create branches
- Filesystem state restoration on branch switch

**Question:** Is this branching system a priority? It adds significant complexity.

### Q3.4: Quasar Tools
Legacy spec 6 tools:
- `quasar_search` - Vector search
- `quasar_add` - Add facts
- `quasar_subtract` - Remove facts
- `quasar_entitize` - Entity graphs
- `quasar_compress` - Summarization
- `quasar_think` - Scratchpad

**Question:** Should we implement all 6, or prioritize specific ones first?

### Q3.5: LGP (Logic Gate Protocol)
Legacy spec defines:
- Logic gates: AND, OR, XOR
- Operators: PIPE, TO
- Example: `{fetch_location} AND PIPE hotels in {location} TO cost_calculator`

**Question:** Is LGP a priority? This is a significant implementation effort.
