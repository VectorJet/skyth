# Questions for Epsilon Specification Review (2026-02-09)

## 1. Implementation Language Mismatch

The current specification (`spec/phase-5/epsilonvcs.md`) details a **Python** implementation (using `zstd`, `hashlib`, Python class structures). However, previous discussions and the Phase 3 `Quasar Architecture` document suggest that core memory components (including Epsilon integration) should be implemented in **Rust** for performance, especially given the "high-frequency ticking" requirement.

- **Question:** Should the Epsilon specification be rewritten to target a Rust implementation (`struct` vs `class`, `Cargo.toml` vs `requirements.txt`, etc.)? If so, should we target a standalone binary or a library embedded in Quasar?

## 2. Integration Mechanics with Quasar

The spec proposes Python-based async hooks (`async def on_quasar_event`). If Quasar is indeed Rust-based, how is this communication intended to happen?

- **Question:**
  - Is Quasar also moving to Python?
  - Or should Epsilon expose a C-ABI / PyO3 binding?
  - Or should Epsilon be a subprocess called via CLI by Quasar?

## 3. Storage Efficiency & Git-Compatibility

The spec proposes a custom Git-like CAS structure (`objects/blobs`, `objects/trees`). While this is "clever as git," it reinvents the wheel.

- **Question:** Have we considered using `libgit2` (via Rust bindings) to leverage existing, battle-tested Git object storage formats directly? This would allow standard Git tools to inspect the repo if needed, while still maintaining our custom "tick" logic on top. Or is the custom format intentional to avoid Git overhead (index, refs, config)?

## 4. Concurrency Model

The Python spec relies on simple `FileLock`. For a high-frequency system potentially accessed by multiple agent threads/processes:

- **Question:** Is `FileLock` sufficient, or do we need a more robust database-style WAL (Write-Ahead Log) or SQLite-based index for the `ticks/` directory to prevent corruption during crashes?

## 5. "Smart Scanning" Bottleneck

The `smart_scanning` algorithm relies on `mtime` and `size`. This is fast but can be unreliable (e.g., rapid modifications within the same second).

- **Question:** Should we incorporate a file watcher (e.g., `notify` crate in Rust / `watchdog` in Python) to maintain a real-time dirty list, rather than scanning the entire directory tree on every tick?
