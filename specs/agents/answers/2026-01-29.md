## A2.2 Agent Calling Agent Architecture 


> Skyth will adopt a hybrid delegation model that supports both inter-agent task delegation and hierarchical sub agent execution.
>
> Concretely, this will be achieved through the introduction of two globally available delegation primitives:
>
> 1. **Agent-to-Agent Task Handoff**, enabling an agent to delegate an isolated task to another peer agent with a distinct role or capability set.
> 2. **Agent-to-Subagent Task Handoff**, enabling an agent to spawn or invoke a subordinate execution context for scoped, task-specific processing.
>
> Both delegation mechanisms will be exposed as **global tools**, accessible to all agents regardless of role, and governed by a unified invocation interface. These tools provide a standardized, explicit mechanism for task transfer while preserving clear execution boundaries, context isolation, and response aggregation semantics.
>
> This dual approach allows Skyth to support both horizontal collaboration between specialized agents and vertical decomposition of complex tasks into sub-tasks, without prematurely constraining the overall agent architecture. Detailed semantics, lifecycle management, and context-passing strategies for these tools are defined in **[[Global Tools]]** (or `@spec/agents/questions/what_are/global_tools.md`).

## A2.3 MCP to TOON Converter

The MCP-to-TOON conversion strategy addresses context window efficiency while maintaining compatibility with existing LLM tool-calling patterns.

**Conversion Flow:**

1. **Tool Invocation (JSON):** LLMs continue to call MCP tools using standard JSON format. This leverages their existing training on tool-use patterns and avoids introducing unfamiliar syntax that could degrade performance.

2. **Tool Output (TOON):** MCP server responses are converted from JSON to TOON before being passed back to the LLM. This reduces token overhead by ~40% for structured data, preventing context window pollution.

3. **Schema Registration (Minimal):** MCP servers are registered with brief tool descriptions rather than full schemas. When an LLM needs detailed information about a tool, it can request schema expansion on-demand.

**Implementation Details:**

- **Converter Location:** The TOON converter operates as middleware between the MCP client and agent runtime
- **Conversion Trigger:** Automatic conversion for all MCP tool results before context injection
- **Fallback Behavior:** If TOON conversion fails (e.g., deeply nested non-uniform data), fall back to JSON
- **When to Convert:** 
  - Always convert tool outputs (results are read-only for LLM)
  - Never convert tool inputs (LLMs generate JSON naturally)
  - Convert large datasets (>100 tokens) for maximum benefit

**Example:**
```python
# LLM calls tool (JSON - native format)
tool_call = {
  "tool": "fetch_users",
  "parameters": {"limit": 100}
}

# MCP returns JSON
mcp_response = {
  "users": [
    {"id": 1, "name": "Alice", "role": "admin"},
    # ... 98 more users
  ]
}

# Converter transforms to TOON before passing to LLM
toon_output = """
users[100]{id,name,role}:
  1,Alice,admin
  2,Bob,user
  ...
"""
# Result: ~40% fewer tokens in context
```

**Rationale:**

- Most models have knowledge cutoffs in 2024 or earlier and lack TOON training data
- JSON tool-calling is a well-established pattern with high reliability
- TOON excels at representing structured output, not input generation
- MCP schema overhead can bloat context; lazy-load schemas only when needed

# Phase 3 LGP AND QUASAR 

## A3.1 LGP Specification

**Logic Gate Protocol (LGP)** provides a structured approach to tool chaining and multi-tool orchestration using logic gate operators.

**Core Design:**

1. **LGP Scope:** Tool chaining and orchestration (Tool A → Tool B → Tool C)
   - Primary use: Sequential tool execution with logical operators
   - Not used for agent-to-agent communication (handled by global tools)

2. **Syntax Design:** 
   - Logic gates: AND, OR, XOR
   - Operators: PIPE, TO
   - Tools called by name without detailed specifications
   - Formalization in Phase 3 with EBNF grammar definition

3. **Tool Execution:**
   - **Shell replacement:** `nu -c` (Nushell) replaces traditional bash tool for better structured data handling
   - **Output format:** All tool outputs received in TOON format for token efficiency
   - **Input format:** Tools invoked using standard JSON parameters

4. **Error Handling:**
   - On chain failure: Pass error message to LLM and continue execution
   - LLM decides whether to retry, modify chain, or abort
   - Partial results from successful steps preserved

**Example:**
```
tool_call: {fetch_location} AND PIPE hotels in {location} AND PIPE {hotel_info} TO cost_calculator
```

**Special Considerations:**

- **quasar_think tool:** Only available for non-reasoning models
  - Reasoning capability detected via models.dev API boolean flag
  - Reasoning models (o1, o3, etc.) bypass quasar_think automatically
  - Non-reasoning models use quasar_think for chain-of-thought processing

## A3.2 Quasar Layer Implementation

**Phase 3 Implementation Scope:**

Quasar will implement Layers 0-3 plus Layer 0.5 (Chroma) by default. Advanced layers (Layer 4: Redis, custom extensions) are deferred to later phases.

**Layer Breakdown:**

- **Layer 0 (Canonical):** Encrypted Jsonl- **Mandatory**
- **Layer 0.5 (Chroma):** Vector database for semantic search - **Default enabled**
- **Layer 1 (JSONL):** Append-only logs - **User configurable**
- **Layer 2 (SQLite):** Relational queries - **User configurable**
- **Layer 3 (PostgreSQL + pgvector):** Advanced semantic search - **Optional, requires user setup**

**Technical Stack:**

1. **Core Implementation:** Rust with Python bindings
   - **Build tool:** Maturin for wheel generation
   - **Distribution:** Pre-built wheels for common platforms
2. **Embedding Models:** 
   - Primary: [TaylorAI/gte-tiny](https://huggingface.co/TaylorAI/gte-tiny)
   - Alternative: `all-MiniLM-L6-v2`
   - Models chosen for lightweight footprint and quality embeddings
3. **Encryption:** 
   - Library: `cryptography` (Python)
   - Storage: Encrypted files in `.skyth/identity/device_auth/`
   - Encryption standard: AES-256

**Layer Interaction:**

- **Write pattern:** Write-through from Layer 0 → Layer 0.5 → Layer 1 → Layer 2 (if enabled)
- **Query cascade:** Query highest available layer first (L3 → L2 → L1 → L0.5 → L0)
- **Data movement:** Real-time writes, batch consolidation for Layer 3 (PostgreSQL)

**Extension Mechanism:**

To add custom layers:
1. Fork main Quasar repository (`@./quasar/`)
2. Implement layer following Rust trait interface
3. Rebuild wheels using Maturin
4. Update `.skyth/quasar/layers.txt` with custom layer numbers
5. **Mandatory layers:** Layer 0 (canonical) only; all others are optional
6. **Conditional layers:** Yes, Layer 3 requires PostgreSQL connection string during setup
7. **Default configuration:** Layer 0 + Layer 0.5 (Chroma) for out-of-box experience

## A3.3 Session vs Daily Storage Logic

**Storage Strategy:** Hybrid daily + session-based memory system running in background processes.

**Session Management:**

1. **Session Creation:**
   - Each session assigned UUID on initialization
   - Session naming: LLM generates human-readable name after receiving first user message
   - Session metadata: UUID maps to LLM-generated name for retrieval

2. **Session Storage:**
   - Location: `.skyth/quasar/md/session/`
   - Format: Markdown files named by session name
   - Metadata: UUID-to-name mapping stored in session index

**Daily Storage:**

1. **Organization Structure:**
```
   .skyth/quasar/md/daily/
   ├── January/
   │   ├── 1st/          # 1st week of January
   │   │   ├── day1.md
   │   │   ├── day2.md
   │   │   └── ...
   │   ├── 2nd/          # 2nd week
   │   └── extras/       # Overflow/misc summaries
   │       └── raw.md
   └── February/
       └── ...
```

2. **Workspace Summaries:**
   - Additional daily summaries: `.skyth/workspace/daily/`
   - Purpose: Task-oriented daily logs separate from conversation memory

**Context Injection:**

1. **New Session Start:**
   - System message: Last day's summary (from `daily/`)
   - System message: Last session summary (from `session/`)
   - System message: Workspace daily summary (from `workspace/daily/`)

2. **Continuing Session:**
   - System message: Last day's summary (even if already in active session)
   - Rationale: Ensures temporal context refreshes daily

**Background Processing:**

- Triggers: After each session ends
- Tasks:
  - Generate session summary (markdown)
  - Update daily aggregation
  - Create embeddings for Quasar Layer 0.5
  - Index session for retrieval

**Multi-Platform Support:**

Reference implementation: `@refs/apps/nanobot`

Supported messaging platforms:
- Telegram
- WhatsApp
- CLI/TUI (primary)
- Web UI (Phase 4)

Each platform maintains unified session tracking via UUID system, ensuring consistent memory across interfaces.

## A3.4 Quasar Event Types and Branching

**Event Type Classification:**

Every tick in the QUASAR system is registered as a quasar event type. This includes:

- **System events:** Background processes, cron jobs, hourly check-ins, heartbeat monitoring
- **User-initiated events:** Standard user messages and responses
- **Branch events:** Solars (user edits) and Nebulas (regenerations)
- **Error recovery events:** API errors, rate limits, timeout handling

**Branching Mechanism:**

**Solars (User Edits):**
Triggered when user edits a previous message. The canonical database is purged up to the Solar sibling branch, and the deactivated branch is sent to create a sibling in other layers.

**Nebulas (Regenerations):**
Triggered by user request or automatic recovery (API errors, rate limits). Similar purge and sibling creation process.

**Naming Convention:**

All events and sibling branches use UUID format with type prefixes:
- `quasar-{uuid}` - Root events
- `solar-{uuid}` - User edit branches
- `nebula-{uuid}` - Regeneration branches

**Branch Hierarchy:**
```
quasar-9f3c7d2a-4e8b-4c1e-b9a5-6b0f1c9d8e42
├── solar-9f3c7d2a-4e8b-4c1e-b9a5-6b0f1c9d8e42      (sibling branch)
│   └── solar-9f3c7d2a-4e8b-4c1e-b9a5-6b0f1c9d8e42+1 (child branch)
└── nebula-9f3c7d2a-4e8b-4c1e-b9a5-6b0f1c9d8e42     (sibling branch)
    └── nebula-9f3c7d2a-4e8b-4c1e-b9a5-6b0f1c9d8e42+1 (child branch)
```

**Branch Increment System:**

- **Sibling branches:** Share parent UUID, stripped of prefix during registration
- **Child branches:** Use `+n` notation where `n` increments with each successive branch
- **UUID format:** `{base-uuid}+{increment-number}`
- **System parsing:** Automatically understands `uuid + '+' + n` structure

**Storage Format:**

All sibling branches are stored as diffs in databases across all layers, enabling efficient storage and reconstruction of branch history.

**CLI Visualization:**

- **Command:** `quasar timeline` displays branch tree structure
- **Edit mode:** Press `Esc` then `e` to enter edit mode and create Solar branches or `r` on a selected AI message for regeneration in the CLI. 
- Both Solar and Nebula events will restore the filesystem in the given folder at that tick Ie. the quasar event, that means restoring deleted files or undoing any changes to the files.
- **Navigation:** Browse through branch history and switch between active branches

**Event Schema Example:**
```json
{
  "event_id": "quasar-9f3c7d2a-4e8b-4c1e-b9a5-6b0f1c9d8e42",
```

## A3.6 Chroma DB + SQL Integration

**Architecture:**

*   **Vector Store:** **ChromaDB** is the default (local, file-based). Production DBs (Postgres/pgvector) are optional.
*   **Relational Store:** **SQLite** (Python native) for metadata and structured queries.

**Data Flow (Parallel Write Strategy):**

*   **Write Path:** Events are written to **all layers in parallel**, not sequentially.
    *   Layer 0 (JSONL) + Layer 0.5 (Chroma) + Layer 1 (SQLite) update simultaneously to minimize latency.
*   **Read Path (CTX):**
    *   **Quasar Context Assembler (CTX)** handles query logic.
    *   Agents do *not* query DBs directly; they request context from CTX.
    *   CTX determines whether to use Vector Search, SQL Filter, or Graph Traversal.

**Agent Interface:**
Agents only need to know the **Tool Names** (`quasar_search`) and **Descriptions**. The underlying complexity (Chroma vs SQLite) is abstracted by CTX.

### Rationale

**1. Core/ Directory:**
   - **Clean root:** Moves backend from root to `core/backend/` for better organization
   - **Future expansion:** `core/` can contain other core components as needed
   - **Clear hierarchy:** Core implementation separated from distribution platforms

**2. Platforms/ Directory:**
   - **Unified client location:** All user-facing platforms in one place
   - **Shared code:** `platforms/shared/` provides TypeScript API types and utilities
   - **Technology grouping:** Groups by role (client platforms) rather than technology
   - **Independent deployment:** Each platform can be built/deployed separately

**3. Separate Repositories:**
   - **Quasar:** Independent Rust project with Python bindings
     - Reason: Large, complex component with its own release cycle
     - Integration: Installed as pip dependency (`quasar-py`) in backend
   - **SUR (Skyth User Repository):** Community marketplace
     - Reason: User-contributed content separate from core codebase
     - Integration: Downloaded/installed via Skyth package manager

### Monorepo Management Strategy

**Three-Tier Approach:**

**1. Just (Primary Interface):**
   - Ergonomic command runner for all development tasks
   - Cross-platform, modern Make alternative
   - Calls into native build tools per language

**Example justfile:**
```justfile
# Development
backend-dev:
    cd core/backend && uv run uvicorn main:app --reload

cli-dev:
    cd platforms/cli && bun run dev

web-dev:
    cd platforms/web && bun run dev

# Building
build-backend:
    cd core/backend && uv build

build-platforms:
    turbo run build --filter="./platforms/*"

build-all: build-backend build-platforms

# Testing
test-backend:
    cd core/backend && uv run pytest

test-platforms:
    turbo run test --filter="./platforms/*"

# Installation
install:
    ./scripts/install.sh
```

**2. Turborepo (Task Orchestration):**
   - Manages TypeScript platforms only (`platforms/`)
   - Provides caching, parallelization, dependency graphs
   - Does NOT manage Python/Rust builds (Just handles those)

**Example turbo.json:**
```json
{
  "$schema": "https://turbo.build/schema.json",
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": ["dist/**", ".next/**", "build/**"]
    },
    "dev": {
      "cache": false,
      "persistent": true
    },
    "test": {
      "dependsOn": ["^build"]
    }
  }
}
```

**3. Native Workspaces Per Language:**

   - **TypeScript (Bun workspaces):**
     ```json
     // platforms/package.json
     {
       "name": "@skyth/platforms",
       "workspaces": ["shared", "cli", "web", "desktop"]
     }
     ```

   - **Python (uv):**
     ```toml
     # core/backend/pyproject.toml
     [project]
     name = "skyth-backend"
     dependencies = ["quasar-py>=0.1.0", ...]
     ```

   - **Rust (Cargo - in Quasar repo):**
     ```toml
     # Quasar/Cargo.toml
     [workspace]
     members = ["quasar-core", "quasar-py"]
     ```

**1. REST API (Standard Operations)**

**2. SSE (Server-Sent Events)**

**Endpoints:**

```python
# - agent_response
# - tool_execution

```

**Event Format:**
```

event: progress_update

```

**3. WebSocket (Bidirectional Communication)**

Purpose: Interactive sessions, real-time collaboration, stateful connections

**Usage:**
- Desktop app (Tauri)

**Connection:**

**Message Format:**


### Authentication Strategy

**Multiple Auth Methods:** Support different client needs

**1. Session-based (Web UI)**
- Login via username + password
- HTTP-only cookies for session management
- CSRF protection
- Suitable for: Web interface

**2. JWT Tokens (CLI, Mobile)**
- Bearer token authentication
- Refresh token mechanism
- Token stored in `~/.skyth/auth/token.json` (CLI)
- Suitable for: CLI, Mobile apps, API integrations

**3. API Keys (Programmatic Access)**
- Long-lived API keys for automation
- Generated via CLI: `skyth auth create-key --name "my-script"`
- Stored in `~/.skyth/auth/api_keys.json`
- Suitable for: Scripts, integrations, CI/CD

**4. Local Auth (Desktop)**
- Superuser password (hashed with Argon2)
- Stored locally, no network auth needed
- Optional: Connect to remote backend with JWT
- Suitable for: Desktop app, local CLI

**Implementation:**
```python
# FastAPI dependency injection
from fastapi import Depends, HTTPException
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials

async def verify_auth(
    credentials: HTTPAuthorizationCredentials = Depends(HTTPBearer(auto_error=False)),
    session: str = Cookie(None),
    api_key: str = Header(None, alias="X-API-Key")
):
    """Support multiple auth methods"""
    if credentials:  # JWT
        return verify_jwt(credentials.credentials)
    elif session:  # Session cookie
        return verify_session(session)
    elif api_key:  # API key
        return verify_api_key(api_key)
    else:
        raise HTTPException(401, "Authentication required")
```

### State Management

**WebSocket State:**
- Maintains active session context
- Tracks conversation history in memory
- Broadcasts updates to connected clients
- Handles disconnection/reconnection gracefully

**SSE State:**
- Stateless (session ID in URL or query param)
- Reconnection via Last-Event-ID header
- Server maintains session state in Redis/memory
- Client responsible for tracking message order

**Session Persistence:**
- All sessions stored in Quasar (permanent)
- Active sessions cached in Redis Layer 2 (if enabled)
- Background processor updates Quasar after session end

**Example State Flow:**
```
User sends message (WebSocket/SSE/REST)
    ↓
Backend validates session
    ↓
Load context from Redis (fast) or Quasar (fallback)
    ↓
Process with agent
    ↓
Stream response (WebSocket/SSE)
    ↓
Update session state (Redis + background Quasar write)
```

### Streaming Method Selection

**Client Capabilities:**

| Platform | Primary | Fallback | Use Case |
|----------|---------|----------|----------|
| CLI | SSE | REST polling | Simple, one-way streaming |
| Web UI | WebSocket | SSE | Interactive, real-time |
| Desktop | WebSocket | SSE | Persistent connection |
| Mobile | WebSocket | REST polling | Battery-efficient |
| Scripts | REST | - | Simple automation |

**Auto-negotiation:**
```python
# Client sends preferred method
GET /api/v1/stream/agent/{session_id}?method=websocket

# Server responds with redirect or upgrade
# If WebSocket supported: Upgrade to WebSocket
# If SSE only: Return SSE stream
# If neither: 406 Not Acceptable
```

---

### Error Handling & Resilience

**WebSocket:**
- Auto-reconnect with exponential backoff
- Resume from last acknowledged message
- Heartbeat/ping-pong to detect dead connections

**SSE:**
- Automatic browser reconnection
- Event ID tracking for resumption
- Graceful degradation to polling if stream fails

**REST:**
- Standard HTTP error codes
- Retry logic in client libraries
- Rate limiting with 429 + Retry-After header

## A9.3: Storage & Version Control (Epsilon)

**Location:** Project-specific `.skyth/` directory (Local Context).

**Epsilon System:**
*   **Definition:** State-based version control system specialized for AI interaction.
*   **Granularity:** Links filesystem states to:
    *   Timestamps
    *   User Messages
    *   AI Responses/Tool Executions
*   **Function:** Enables "Time Travel" (Solars/Nebulas) by reverting project state to specific interaction ticks.
*   **Difference from Git:** Automated, granular, metadata-rich, intended for session replay/branching rather than code versioning.
